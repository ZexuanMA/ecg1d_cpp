N=2 ECG/TDVP Simulation Problem Analysis

This report analyzes why the current code behaves well for N=1,K=3 but becomes suspicious for N=2. The strongest evidence points to real-time TDVP instability in a very rank-deficient N=2 tangent space, plus several verification and observable issues that can hide or mislabel the failure.

Executive Diagnosis

The static N=2 ground-state part is probably mostly correct for the smooth noninteracting T + V_ho + V_kick benchmark: out/verify/step3_crosscheck_N2.csv shows ECG energy agreement with grid/DVR at about 1e-5 to 8e-5. The real-time N=2 part is not in the same condition. Existing logs show a large violation of structural conservation after real-time TDVP starts.

TDVP Equations

The intended real-time projected equation is:

\[ |\dot\psi\rangle = \sum_{\beta}\dot z_\beta|\partial_\beta\psi\rangle, \qquad |R\rangle=i|\dot\psi\rangle-H|\psi\rangle . \] \[ \langle\partial_\alpha\psi|R\rangle=0 \quad\Longrightarrow\quad i\,C_{\alpha\beta}\dot z_\beta=g_\alpha, \qquad C_{\alpha\beta}=\langle\partial_\alpha\psi|\partial_\beta\psi\rangle, \qquad g_\alpha=\langle\partial_\alpha\psi|H|\psi\rangle . \]
The implementation uses a normalized/covariance form of \(C\), which is fine if the same projection is used consistently and the solved system stays Hermitian positive semidefinite on the tangent space.

Ranked Problems

HighThe N=2 real-time tangent metric is extremely rank deficient.

In Step 1 for N=2,K=7, the imaginary-time log reports cond(C) around 2.3e9 to 3.1e9 and effective rank 7. The realtime parameter count is \(K(1+N+N+N(N+1)/2)=56\), so most directions are gauge-like or numerically redundant.

The real-time RHS is solved with an SVD pseudoinverse in src/realtime_tdvp.cpp:26-73. In this situation tiny singular-mode changes can cause large motion in parameters without corresponding physical motion. This matches the log pattern: |dz| grows while norm and raw energy drift.

Probable fix direction: remove redundant coordinates first. For a harmonic free-evolution check, freeze A,B,R and evolve only u with the fixed-basis generalized Schrodinger equation, or use --u-split-trotter plus a much smaller geometry step. For full TDVP, explicitly project out normalization, phase, and near-null gauge directions before solving.

HighRegularized real-time TDVP may be breaking the Hamiltonian projection structure.

The code assembles \(C\), then solves with C_bar = C_mat.conjugate().transpose() and adds lambda_C I directly before SVD. See src/realtime_tdvp.cpp:26-32.

For real-time TDVP, conservation depends on solving a Hermitian projected equation. If the analytic derivatives make \(C\) slightly non-Hermitian, or if regularization/truncation is not applied in a structure-preserving way, norm and \(E_{\mathrm{raw}}=\langle\psi|H|\psi\rangle\) can drift even when normalized energy looks nearly flat.

Quick check: print \(\|C-C^\dagger\|/\|C\|\), eigenvalues of \((C+C^\dagger)/2\), and \(\mathrm{Im}(g^\dagger C^+g)\) at every step for N=2. If \(C\) is not Hermitian positive semidefinite, solve the symmetrized metric only after validating the derivative formulas.

HighStep 4 has no N=2 reference dynamics, so failures are not localized.

main_verification.cpp:269-287 explicitly skips grid and DVR reference dynamics when N>=2. For the current no-interaction N=2 verification, the many-body state should factor into two identical one-body evolutions, so an N=2 reference is cheap and should exist.

Without this reference, the code can only report internal conservation diagnostics. It cannot tell whether density, \(n(k)\), fidelity, or energy are physically correct after the N=2 TDVP path starts drifting.

Fix direction: reuse the N=1 grid/DVR propagator and map observables as \(E_{N=2}=2E_{1p}\), \(\langle x^2\rangle_{1p}\), \(\langle p^2\rangle_{1p}\), and density per particle. Then add step4_cross_check support for N=2 noninteracting product states.

MediumStep 4 evolves only \(T+V_{\mathrm{ho}}\), not the full kicked Hamiltonian.

src/verify/step4_realtime_tdvp.cpp:79 hardcodes HamiltonianTerms::kinetic_harmonic(). Step 1 prepares the ground state of \(T+V_{\mathrm{ho}}+V_{\mathrm{kick}}\), but Step 4 turns off the cosine potential. This is valid as a harmonic free-evolution test, but it is not a kicked-rotor or interacting N=2 simulation.

If the expected N=2 behavior includes kicks or contact interaction, this setup is testing the wrong Hamiltonian. The first check should state the exact pair \(H_{\mathrm{prepare}}\) and \(H_{\mathrm{evolve}}\) in the log.

Medium\(\langle x\rangle=\langle p\rangle=0\) in N=2 is expected, not necessarily a bug.

The N=2 SVM path initializes B=0 and R=0 in src/svm.cpp:285-289 and src/svm.cpp:314-318. Under the parity-even Step-4 Hamiltonian, the state remains parity even. Therefore odd observables such as \(\langle x\rangle\) and \(\langle p\rangle\) should stay exactly zero.

This explains the zero traces, but it does not explain norm drift. To test visible center-of-mass motion, start from a small parity-broken displacement such as \(R_0=R_1=d\), then verify harmonic oscillation.

MediumN=2 momentum density is not generally implemented.

src/observables.hpp:10-16 warns that momentum_distribution is a form factor, not the true momentum probability density. In Step 4, src/verify/step4_realtime_tdvp.cpp:120-127 uses momentum_density_from_density for N>=2.

That reconstruction is only valid when the many-body state is a noninteracting product state with a real, nonnegative one-body orbital. It is not valid for interacting N=2, for kicked states with phase structure, or for correlated ECG states. Any N=2 \(n(k)\) plot after nontrivial dynamics is therefore a probable observable artifact.

Fix direction: implement the one-body reduced density matrix \(\rho_1(x,x')\) and compute \(n(k)=(2\pi)^{-1}\int dx\,dx'\,e^{-ik(x-x')}\rho_1(x,x')\).

MediumObservable sampling ignores the requested Hamiltonian terms.

sample_observables accepts terms, but src/realtime_tdvp.cpp:293-299 computes only kinetic plus harmonic energy. This is harmless for the current Step-4 free harmonic check, but wrong for any realtime run with delta, gaussian, or kicking enabled.

Fix direction: use compute_total_energy(basis, terms) for reported energy, and separately write components with compute_energy_components.

MediumThe ideal kick path is step-size dependent if used as a TDVP step.

src/kicked_evolution.cpp:132-136 applies an ideal delta kick as one realtime TDVP step with dt_step. An ideal kick should apply a finite unitary phase independent of an arbitrary numerical step:

\[ U_{\mathrm{kick}} = \exp\left[-i\kappa\sum_{a=1}^{N}\cos(2k_Lx_a)\right]. \]

If your N=2 simulation uses the kicked-evolution path, this can be a major derivation/code mismatch. Prefer the analytic kick projection, or a Rothe/target-fitting step that fits \(U_{\mathrm{kick}}|\psi\rangle\) directly.

MediumN=2 target-fitting/kick expansion is incomplete.

src/target_fitting.cpp:46-50 throws for N!=1. This means the clean target-state approach is not yet available for N=2. If current N=2 kicked dynamics relies on tangent TDVP instead, it is using the more fragile route.

LowerPairCache complex Gaussian branch and health checks may fail late in realtime.

PairCache evaluates \(M_G\) through a complex determinant square root. During real-time evolution, complex widths can move near singular or branch-sensitive regions. The current verbose health check mainly prints diagonal Re(A+B), which does not guarantee that every pair/permutation \(K\) has a safe positive real part.

Add monitors for the minimum eigenvalue of \(\mathrm{Re}\,K\), \(|\det K|\), and the phase of \(\det K\) over all active pairs and permutations.

LowerContact interaction may be a basis limitation, not a local bug.

Prior notes in past_md/N2_summary.md report that the N=2 delta benchmark stalls near 2e-3 and TDVP becomes unstable because a Gaussian basis converges poorly to the contact cusp. If the problematic N=2 run includes \(g\delta(x_1-x_2)\), expect basis incompleteness and metric conditioning to dominate even after code bugs are removed.

Recommended Debug Order

  1. Build an N=2 noninteracting product-state reference for Step 4. Do this before changing TDVP internals, so every later change has a physical target.
  2. Run a short-time diagnostic with fixed geometry and exact \(u\)-evolution. This should conserve norm and energy and match the product reference.
  3. For full geometry TDVP, print \(C\) Hermiticity, \(C\) eigenvalues, effective rank, and local \(d\langle\psi|\psi\rangle/dt\), \(d\langle H\rangle/dt\) predicted by the RHS before each step.
  4. Reduce the coordinate set: first \(u\) only, then \(u+A\), then \(u+A+R\), then full \(u+A+B+R\). The first coordinate set that breaks conservation is the likely derivation/code surface.
  5. Replace N=2 \(n(k)\) plots with either the product-state reference momentum density or a real one-body-density-matrix implementation.
  6. If the final goal is kicks, implement N=2 analytic kick projection or Rothe fitting instead of treating the ideal kick as one finite TDVP step.

Bottom Line

The most probable root cause is not the static N=2 Gaussian matrix elements. It is the real-time TDVP solve on an overparameterized, nearly singular N=2 tangent space. The current verification pipeline then makes the situation harder to diagnose because N=2 Step 4 lacks a reference solver and reports some observables that are either symmetry-protected zeros or only approximate reconstructions.
