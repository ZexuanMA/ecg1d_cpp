ECG1D main.cpp Principle and Mathematical Review

This document translates the execution logic of main.cpp and its numerical backend into mathematical language. The emphasis is on what the program actually computes: a finite-dimensional explicitly correlated Gaussian manifold, generalized eigenvalue optimization, TDVP evolution, and kicked-dynamics projection in a non-orthogonal basis.

Main Thesis

The code represents the many-body state as a finite non-orthogonal Gaussian expansion Ψ(x) = Σk ukΦk(x). Every major routine in main.cpp either improves the manifold {Φk}, updates the coefficient vector u, or propagates that state under a chosen Hamiltonian.

What Is Reviewed

The explanation below covers the implemented mathematics, not only the intended theory. Where the code contains a material modeling or numerical risk, it is called out explicitly in the review appendix.

1. High-Level Overview

main.cpp is an orchestration layer. The real mathematics lives in the modules that assemble overlap kernels, Hamiltonian matrix elements, overlap derivatives, TDVP metric tensors, and kick operators. At the program level, the code follows one of two archetypes:

Static Problem

Build a non-orthogonal ECG basis, assemble H and S, solve Hc = ESc, then improve the basis by stochastic search and imaginary-time TDVP.

Dynamic Problem

Starting from a variationally optimized state, evolve either by real-time TDVP or by the more stable fixed-basis strategy: analytic kick projection plus exact generalized-eigenbasis free propagation.

1

State Construction

Read or synthesize basis parameters (u, A, B, R) and interpret them as correlated Gaussian orbitals.

2

Matrix Assembly

For every pair of basis functions, compute Gaussian pair integrals and assemble the overlap and Hamiltonian matrices.

3

Variational Solve

Solve a generalized Hermitian eigenproblem to obtain the lowest energy and its coefficient vector in the current basis.

4

Evolution or Diagnostics

Apply TDVP, exact fixed-basis propagation, or analytic kick projection depending on the chosen execution path.

2. Variational State and Basis Ansatz

The finite-dimensional trial state is a linear combination of explicitly correlated Gaussian basis functions:

Ψ(x) = Σk=1K uk Φk(x)

For each basis element, the parameter set stored in BasisParams is (u, A, B, R, name). The nonlinear part of the basis function is of the form

φk(x) = exp[-xT(Ak + Bk)x + 2 RkT Bk x - RkT Bk Rk]

Interpretation of Parameters

  • A is a complex symmetric correlation matrix controlling the quadratic core of the Gaussian.
  • B is implemented as diagonal and acts as an additional width operator, especially useful for dynamics.
  • R is a complex displacement-like vector. Its imaginary part encodes momentum-carrying phases.
  • u is the linear coefficient in the global expansion, not part of the orbital shape itself.

Momentum Encoding

If Baa = b and Ra = i q / (2b), then the linear term becomes

exp[2 Ra Baa xa] = exp[i q xa]

This is why the kick-related routines generate additional basis functions by assigning purely imaginary components to R.

Implemented symmetry convention. Matrix elements are always summed over permutations with weights sP = perms.signs[p]. In the current implementation, sP = 1 for all permutations, so the actual code realizes a bosonic symmetrized sum rather than a generic signed permutation sum.

3. Gaussian Pair Cache and Operator Kernels

All matrix elements reduce to pairwise Gaussian integrals. For a bra basis function i, a ket basis function j, and a permutation matrix P, the code constructs a cached object PairCache containing the effective quadratic form:

K = Ai* + Bi* + PTAjP + PTBjP
bT = 2 Ri*T Bi* + 2 RjT Bj P
C = -Ri*T Bi* Ri* - RjT Bj Rj, μ = (1/2) K-1 b
MG = πN/2 det(K)-1/2 exp[C + (1/4)bTK-1b]

This scalar MG is the fundamental Gaussian overlap kernel. Every physical matrix element is obtained by multiplying MG by an operator-specific kernel.

Overlap Kernel

Sij = ΣP sP MG(P)

Hamiltonian Kernel

Hij = ΣP sP MG(P) Kop(P), where Kop is the local operator kernel.

Term Implemented Kernel Meaning
Kinetic P = -2 tr(KMj) + gTg - 4 gTKMjμ + 4 μTKMj2μ + 2 tr(KMj2K-1) Contraction of the Laplacian against the Gaussian pair.
Harmonic Trap rTr = (1/2) tr(K-1) + μTμ Expectation of x2 inside the Gaussian pair.

Hamiltonian implemented in code. With the active switches in HamiltonianTerms, the solver evaluates H = T + Vho + Vδ + Vgauss + Vkick, with the default constants ℏ = m = ω = 1.

4. Generalized Eigenproblem and Energy Functional

Because the ECG basis is non-orthogonal, the variational problem is not a standard eigenproblem. The coefficient vector is determined by the generalized Hermitian system

H u = E S u

Equivalently, the energy of a state written in the current basis is the Rayleigh quotient

E[u] = (u† H u) / (u† S u)

Numerically, the code diagonalizes S, truncates numerically unsafe eigenvectors, builds S-1/2, and solves the transformed Hermitian problem

H̃ c = E c, with H̃ = S-1/2 H S-1/2

Why This Matters

  • The overlap matrix can become ill-conditioned when basis functions are nearly linearly dependent.
  • Truncation in the small-eigenvalue subspace prevents uncontrolled amplification by S-1/2.
  • The code cross-checks the lowest eigenvalue against the corresponding Rayleigh quotient before accepting it.

Additional Diagnostic

The solver also computes an energy-variance proxy σ2 = ⟨H2⟩ - E2. In a perfect eigenstate this should approach zero, so it functions as a quality certificate for the current basis.

5. SVM Construction and Stochastic Refinement

The first part of the pipeline is basis discovery. The code does not assume an optimal nonlinear manifold in advance. Instead it builds one by a stochastic variational method, then refines it by local perturbations.

BK+1 = arg minφ in candidate pool E0(BK ∪ {φ})
Greedy SVM Stage

svm_build_basis() samples trial ECGs, screens out near-duplicate functions by normalized overlap, and accepts the candidate that produces the smallest lowest generalized eigenvalue.

Refinement Stage

stochastic_refine() perturbs one basis function at a time, rebuilds H and S, and keeps a replacement only if the ground-state energy decreases.

Mathematical role of SVM. The linear coefficients solve only a finite-dimensional problem once the manifold is fixed. The difficult part is the nonlinear search over the manifold itself. SVM is the code's practical surrogate for the infinite-dimensional minimization infmanifold infu != 0 (u†Hu)/(u†Su).

6. TDVP as a Riemannian Evolution Equation

Once the basis parameters are treated as dynamical coordinates z = {u, B, R, A}, the code constructs the overlap-geometry matrix C and the Hamiltonian gradient g.

Cαβ = (1/S) ∂2S / (∂zα ∂z̄β) - (1/S2) (∂S / ∂zα) (∂S / ∂z̄β)
gα = ∂E / ∂z̄α

This is precisely the metric-like tensor implemented by calculate_C(). In geometric terms, the code computes the projected equation of motion on a non-orthogonal variational manifold.

Imaginary-Time TDVP

C dz = -g

The solver regularizes C with a Tikhonov term, solves by truncated SVD, applies a line search, and accepts a step only if the energy decreases. In this mode, the code excludes the u coordinates from the update and may re-solve them from the generalized eigenproblem.

Real-Time TDVP

C dz/dt = -i g

kicked_evolution.cpp uses the real-time version and integrates it by a classical RK4 scheme. In this branch the coefficient variables u are included in the update.

Derivative architecture. The derivative layer is exact at the analytic kernel level: first derivatives of the logarithmic Gaussian overlap generate the metric, and Hamiltonian gradients are decomposed into d(MG) and d(kernel) parts. That is why the TDVP implementation looks algebraically dense: it is carrying closed-form derivatives, not finite differences.

7. Kicked Dynamics: Projection, Basis Augmentation, and Exact Free Propagation

The kicked branch is mathematically the most interesting part of the code. It mixes three ideas: momentum-aware basis augmentation, analytic kick projection, and exact fixed-basis free evolution.

7.1 Kick Operator

Ukick = exp[-i κ Σa=1N cos(2 kL xa)]

The code uses the Jacobi-Anger expansion

exp[-i κ cos θ] = Σn=-∞∞ (-i)n Jn(κ) exp(i n θ)

Because the matrix element of exp(i n 2kL xa) against a Gaussian pair is analytic, the kick matrix can be assembled explicitly:

Kkickij = ⟨Φi|Ukick|Φj⟩
u' = S-1 Kkick u

7.2 Exact Fixed-Basis Free Evolution

Between kicks, the most stable branch keeps the basis fixed and propagates only the coefficient vector in the generalized eigenbasis:

u(t) = S-1/2 exp[-i H̃ t] S1/2 u(0)

This avoids stepwise TDVP drift during the free interval. Mathematically it is an exact propagator inside the currently chosen finite basis.

7.3 Why Basis Augmentation Is Necessary

A kick does not preserve the original Gaussian manifold centered around the ground state. The phase factor exp(i n 2kL x) generates shifted momentum sectors, so the code augments the basis with momentum-carrying ECGs.

Baug = Bgs ∪ Bmom

In the --kicked-gamma0 path, the code also scores repair packets by a first-kick objective:

J = rel(E) + 3 |Δ⟨x2⟩| + rel(⟨p2⟩) + 0.25 |1 - fidelity| + 5 penaltygs + 0.05 penaltycond

This objective is not abstract theory; it is literally the acceptance criterion used to improve the post-kick representation quality while still protecting overlap conditioning.

Interpretation of the kicked branch. The code is not merely solving a TDVP ODE. It is trying to build a finite subspace that is approximately invariant under alternating actions of Ukick and exp(-i Hfree T).

8. What the Main Modes Actually Mean

The command-line flags in main.cpp are not cosmetic. Each mode corresponds to a distinct numerical experiment.

Default Path

Reads or constructs basis functions and prints functional and derivative diagnostics. This is primarily a kernel-validation mode.

--tdvp / --tdvp-only

Tests the TDVP machinery, including the one-particle harmonic benchmark where Gaussian evolution should be especially favorable.

--delta, --gaussian, --kicking

Run the full static chain: SVM basis construction, stochastic refinement, then imaginary-time TDVP improvement for different Hamiltonian choices.

--kicked

Executes real-time kicked evolution with the TDVP machinery, intended as a direct dynamical experiment rather than a stationary solver.

--kicked-gamma0

Uses a more carefully engineered route: build a ground-state basis, augment it with momentum sectors, apply analytic kick projection, and evolve exactly in the fixed augmented basis.

--kick-convergence

Studies the finite-pulse TDVP approximation as T_pulse → 0 and compares it with the exact instantaneous-kick solution.

9. Review Findings

These are the main correctness and design issues I found while reviewing the code path around main.cpp.

High

The “ideal delta kick” in the TDVP path is step-size dependent.

In main.cpp, run_kicked_evolution_test() sets T_pulse = 0.0 and then calls kicked_evolution(). In apply_kick(), the T_pulse <= 0 branch performs exactly one realtime_tdvp_step(..., dt_step, terms_with_kick, ...).

Mathematically, that applies a finite-duration TDVP update of size dt_step, not an impulse with fixed area κ. Since the kick Hamiltonian itself uses the unscaled constant amplitude from physical_constants.hpp, the supposed delta kick is effectively proportional to the integrator step size. The physics therefore changes if dt_step changes.

Medium

KickParams.kappa_kick and KickParams.k_L_kick are dead configuration fields.

The public API advertises per-run kick strength and lattice wavevector inside KickParams, but the kicked TDVP implementation never uses them. The actual kick amplitude and wavevector come from the global constants kappa and k_L, with only kicking_scale available as a multiplier.

As a result, callers can set kick_params.kappa_kick or kick_params.k_L_kick and observe no effect. That is a silent API mismatch and a likely source of misleading experiments.

Scope note. The permutation framework stores a sign for each permutation, but the current implementation sets every sign to +1. For your present bosonic cold-atom use case this may be intentional. If you later intend to support antisymmetric states, that layer is not yet doing it.
